Mapping tables

A mapping table is a table loaded in the Data load editor with a MAPPING prefix (MAPPING LOAD or MAPPING SELECT).

Mapping tables are treated differently from other tables:

  • A mapping table must have a table name and only two columns (fields).
  • The key field must be loaded first in the MAPPING LOAD statement.
  • The two fields in the mapping table must be named, but the names are not relevant. Instead, the order of the fields is relevant for the mapping. The name of the mapping table is also relevant for the mapping.
  • The mapping table must be loaded earlier in the script before the declaration of the table/tables where you apply the mapping.
  • The ApplyMap function is used in other tables to map the values from the mapping table. Optionally, a default value can be defined if no matches are found in the mapping table:

    ApplyMap('map_name', expression [, default_mapping ])

  • When mapping a certain field value or expression, that value is compared to the values in the first field of the mapping table. If found, the original value is replaced by the corresponding value in the second field of the mapping table. If not found, the default value is used. If a default value is not specified, no replacement is made.
  • After the script execution, these mapping tables are automatically dropped. They just exist during script execution.
Note: Be aware that Qlik Sense does not generate a script error if the mapping is unable to find any matches when using ApplyMap. It is always a good practice to check the values generated after using ApplyMap.
Mapping table syntax example

Statement for creating a mapping table:

Country_Map:
MAPPING LOADCountryCode,CountryName;SQL SELECT * FROM CountryTable;

The following shows the ApplyMap line added to the table you map to.

ApplyMap('Country_Map',CountryID) AS Country

OR

ApplyMap('Country_Map',CountryID,'Other') AS Country

CountryID is a field from the table you map to. It contains the reference values to look up against the first field, CountryCode, in the mapping table. These field names can be different.

If the mapping is not working as expected, check the following:

  • Does the mapping table have the MAPPING prefix before LOAD or SELECT ?
  • Does the mapping table have a name, two fields with names, and is declared before applying the mapping with ApplyMap?
  • Is the order of the fields in the mapping table correct?
  • Is the ApplyMap function using the correct mapping table name and the correct field name/expression?
  • Are the values in the first field of the mapping table the same format as the values in the field using the ApplyMap function?

Preceding Load on preceding Load

One of the useful features of Qlik Sense is the ability to perform preceding LOAD on preceding LOAD. Additional preceding loads perform calculations on the output of another preceding LOAD.

Using this technique is a great way of transforming your table even further.

Preceding Load on preceding Load example

Load A, B, and calculated fields X and Y are loaded in the succeeding SELECT statement from Table1 . From all the fields in the preceding LOAD, two are combined, A and X, to create a new field E.

Notice that the two LOAD statements refer to the same SELECT statement. Qlik Sense will execute the first one above the SELECT statement. Once this is complete, it will execute the preceding LOAD statement. It will keep trying to find preceding LOAD statements until it reaches to the top of the table.

LOAD *, A & X AS E;
LOAD A, B, if(C>0, 'positive', 'negative') AS X, weekday(D) AS Y;
SELECT A, B, C, D FROM Table1;

Joining tables

A JOIN is an operation that is applied to two tables and combines them into one.The final table contains the combination of records from the original tables and all the fields from both tables.

The JOIN statement is defined before the LOAD or SELECT statement of the second table. It may be preceded by one of the following prefixes:

  • INNER — The final table only contains records presented in both tables.
  • OUTER — All combinations between the two tables are generated. The final table contains all the records from both tables.
  • LEFT — The final table only contains records presented in the first table.
  • RIGHT — The final table only contains records in the second table.

JOIN consolidates data from multiple tables into a single table based on the same field names between two tables. If multiple field names exist, all values in those fields must match, or new records can be generated depending on the JOIN type used.

Pros:

  • Joins are good for adding multiple fields onto existing records.
  • LEFT, RIGHT, OUTER and INNER prefixes can be used to limit data.
  • Joins are faster than ApplyMap functions.
  • Joins can handle one-to-many relationships.

Cons:

  • You can unintentionally generate new data records or duplication. This is especially dangerous in fact tables containing key measures.
  • You cannot specify an alternative value if no match exists between tables (as you can with the third parameter of the ApplyMap function).
Note: By default, if no prefix is specified, the Outer prefix is applied. A bad Join can result in significant memory penalties. In particular, if no field names match, a Cartesian product of the two tables is created when using an Outer Join and causes an explosion of row numbers.
Example of a LEFT JOIN between two tables
LOAD a, b, c
FROM Table1.csv;
LEFT JOIN (Table1)
LOAD a, c, d, e
FROM Table2.csv;

The result is a single table called Table1 containing a, b, c, d, and e fields, and only the records from the original Table1. The records for d and e fields are only populated from Table2 if the values in the common fields (a and c) exist in Table1.

Concatenation

Concatenation is another way to combine tables. It appends data from multiple tables into a single table.

There are two ways to concatenate:

Automatic concatenation

Qlik Sense automatically concatenates the results of the different LOAD or SELECT statements into one table if the field names and the number of fields within two or more tables are the same:

LOAD a, b, c FROM Table1.csv;
LOAD a, c, b FROM Table2.csv;
The result is a single table (Table1) with a, b, and c fields and all the records from Table1 and Table2.

Forced concatenation

If two or more tables do not have exactly the same set of fields, it is still possible to force the concatenation using the CONCATENATE prefix in front of the LOAD or SELECT statement.

LOAD a, b, c FROM Table1.csv;
CONCATENATE
LOAD a, c FROM Table2.csv;

The result is a table called Table1 with the a, b, and c fields and all the records of Table1 and Table2. Some b values are NULL.

Preventing concatenation

If two tables have the same set of fields, which is usually automatically concatenated, you can prevent the concatenation with the NoConcatenate prefix.

This statement prevents concatenation with any existing table in the data model, with exactly the same fields.

The syntax is: 

NOCONCATENATE ( LoadStatement | SelectStatement )
Example
LOAD a, b, c FROM Table1.csv;
NOCONCATENATE
LOAD a, b, c FROM Table2.csv;

The result is two tables called Table1 and Table2, each with the a, b, and c fields.

Just like all the other transformation techniques, concatenation has its pros and cons.

Pros:

  • Concatenation is good for combining multiple tables without the risk of generating new records.
  • Concatenated data is easy to code and maintain, even as data changes.
  • Concatenation uses less processing power than JOIN.

Cons:

  • No efficiencies are gained by combining fields, as in a JOIN.
  • NULL values can be created after concatenating tables.
  • CONCATENATE can generate larger tables than JOIN.

Concatenation rules

  • Common fields for all tables must have exactly the same name.
  • The fields can be listed in any order in each statement.
  • Unless a table name of a previously loaded table is specified in brackets immediately after the CONCATENATE statement, the CONCATENATE prefix uses the last previously created internal table. Therefore, with forced concatenation, the order of the statements can matter.
  • To specify a table to concatenate with, but not immediately follow, put the table name in brackets. For the preceding example, you can use:
    CONCATENATE (Table1) LOAD a, c FROM Table2.csv;
