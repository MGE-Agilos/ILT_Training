Defining and working with data sets in expressions

Controlling selections at the expression level in a Qlik Sense app is possible using a specific syntax in the expression, also known as set analysis syntax.

Qlik allows users to analyze sets of data in an app not driven exclusively by user selections, but by other dynamic or static sets defined using set analysis.

Set analysis

Set analysis is a feature in Qlik Sense useful for:

  • Providing power to Qlik Sense apps by controlling the data set that is being evaluated for each expression.
  • Allowing dynamic solutions to cut down on the amount of scripting that might otherwise be required.

  • Improving performance over the use of an IF statement.

Set expressions

A set expression is a set of field values defined in a statement. Use set expressions to work with set analysis.

Set expressions are:

  • Only used in the context of an expression.
  • Not available to use in the data load script.
  • Composed of three elements, but not all parts have to be present.

The three parts of a set statement:

  • Set identifier (can be omitted in some cases)
  • Set modifier (optional)
  • Set operator (optional)

For more information, search Qlik Help pages for Set analysis and set expressions F.

Set identifiers

The set identifier defines how the set of field values in the app relates to the field or expression you are evaluating.

Examples:

Set identifier Example Result
$ Sum({$} Sales) Returns sales for the current selection, so it is the same as Sum(Sales). This identifier can be omitted.
1 Sum({1} Sales)

Returns total sales within the application, disregarding the selection, but not the dimension.
$1 Sum({$1} Sales) Returns sales for the previous selection.

For more information, search Qlik Help pages for Syntax for sets F.

You can also use Bookmarks and Alternate states as set identifiers:

  • Use the Alternate state name in set analysis: Sum( {State2} Sales)
  • Use the Bookmark name in set analysis: Sum( {MyBookmark} Sales)

Set modifiers

Modify a set by making an additional or changed selection. The modification is written in the set expression.

The modifier consists of one or several field names, each followed by a selection to be made on the field, all enclosed by angle brackets < and >.

Examples:

Sum( {1<Region= {'USA'} >} Sales )

Returns the sales for the region 'USA', disregarding the current selection.

The curly brackets define the element set, where the elements can be either field values or searches of field values.

Sum( {$<Year = {2016}, Region = {'USA', 'SEW','UK'}>} Sales )

Returns the sales for the current selection, but with the new selections both in Year and in Region.

When multiple field values are listed within curly brackets, the values are separated by commas.

Sum( {$<Year = {">2013<=2016"}, Division = {'North America'}>} Sales )

Numeric search that specifies a defined range applied in numeric fields using double quotes, as in the Year field of this example.

Sum( {<Division= {"*America"}>} Sales )

String search that is applied using double quotes and wildcard characters, such as an asterisk or question mark.

This expression returns the sales for all the divisions containing America at the end of the division name, such as 'North America' or 'South America'.

Notice how the set identifier has been omitted. As a result, the selections in other fields will apply to the calculation.

Sum( {$<Year =- {"<2010"}>} Sales )

Set operators can be applied to sets. In this case, the exclusion (-) is used to exclude all the years before 2010 from the available years.

Sum( {$<Year = {2016}, Month= ,Quarter= >} Sales )

This expression specifies fields in the set modifier without a set; all the values in those fields are included in the calculations. The selections in them are not affecting the result.

Sum( {$<Year = {$(=YEAR(TODAY())-3)}>} Sales )

Returns the Sales for the year: current year - 3.

Dynamic expressions can also be used in set analysis. They must always be enclosed in a dollar-sign expansion syntax and begin with =.

Note: If no set identifier is specified, the $ is assumed. It is a best practice to specify a set identifier.

For more information, search Qlik Help pages for Set modifiers F.

Set operators

Use set operators to create new sets based on the union (+), exclusion (-), intersection (*), and symmetric difference (/), also known as mutual exclusion.

Examples:

Union (+)

Year={">2015"}+{2010}

Any year after 2015 and 2010

Exclusion (-)

Year=-{"<2010"}

Any years except years before 2010

Intersection (*)

Year={">2014"}*{"<=2016"}

Only 2015 and 2016 years

Symmetric difference (/)

Year={">2014"}/{"<=2016"}

Any year after 2014 or before 2016

Aggregation functions

Aggregation functions, by default, aggregate over the current selections.

These functions, like Sum() F, Max() F, Min() F, or Avg() F, use the set of possible records defined by the selection. Then, they return a single value based on the calculation defined by function, like a sum, an average, or a count.

Example:

Sum(LineSalesAmount)

A field without any function always represents the array of possible values. However, applying a function to a field performs a calculation on them:

  • OrderDate: An array of values in the field
  • Max(OrderDate): A single value calculating the maximum value of the OrderDate field

A field in an expression returns NULL unless there is only one possible value; if that is the case, it returns that value.

The Only function works in a similar way:

  • If the field returns several values, the function interprets it as NULL.
  • If the field has one possible value, then the function returns that value.

Using this function, instead of using just the field, has the advantage of being able to use set analysis.

Example:

=Only({$<UnitPrice={9}>} Product)

Defining the aggregation scope

There are usually two factors that together determine which records are used to define the aggregation value in an expression. When working in visualizations, these factors are:

  • Dimensional value (of the aggregation in a chart expression)
  • Selections

Together, these factors define the scope of the aggregation. You may come across situations where you want your calculation to disregard the selection, the dimension or both.

In chart functions, you can achieve this by using the Total qualifier, All qualifier, or set analysis.

Total qualifier

Using the Total qualifier inside your aggregation function disregards the dimensional value.

All qualifier

Using the All qualifier inside your aggregation function disregards the selection and the dimensions.

Note: The equivalent can be achieved by using a combination of {1} and Total qualifier:

=sum({1} Total Sales)

{1} set analysis identifier

Using the {1} set analysis identifier will ignore the user selection and always evaluate the entire data set.

Aggr function

The Aggr function is typically used in charts to extend the aggregation of a given expression or to create calculated dimensions. It is also used outside of the context of a chart to retrieve aggregated values.

Aggr(([ Distinct | NoDistinct ] [{set_expression}] expression {, Dimension}))

The Aggr function is to the front-end, what the Group By is to the script.

Example:

Assume you have the following table, which can be considered a simplified version of your OrderDetails table in your data model:

OrderID LineNo LineSalesAmount
1 1 30
1 2 20
1 3 50
2 1 10
2 2 20
3 1 40

The above table is an order-line level table, which means that you have one row for every order line and may have multiple rows for the same OrderID.

You want to create a list of aggregated values that hold the total sales amount for every order (Sum(LineSalesAmount) per OrderID). This new table has one row for every OrderID value, like below:

Sum(LineSalesAmount)
100
30
40

To create this table, use the Aggr function in the following way:

=Aggr(Sum(LineSalesAmount), OrderID)

This expression creates a virtual table in memory, with the values that you see in the new table.

  • If the Aggr function is used in the case of a calculated dimension, then the values on this list are the dimension values.
  • If the Aggr function is used inside an aggregation function, then the expression will return one single value:

    =Max(Aggr(Sum(LineSalesAmount), OrderID)) returns 100

    =Min(Aggr(Sum(LineSalesAmount), OrderID)) returns 30

Reusable items and object library

After the data is loaded in the app, you can define frequently used fields and calculations and store them as master dimensions and measures. There are also other objects that can be created and be reused in the app, such as app charts or visualizations, fields, and selection states.

  • These dimensions and measures can be used in several visualization and objects across the app, but they are maintained or modified from one single entry point, in the Master items area.
  • Master items and sheet assets are found in the Assets panel of the sheet editor.

Master items

Reusable fields, calculations, and visualizations can be set or defined in the app. They are called Master items and allow the users to apply global changes to visualizations, dimensions, and measures in one app.

Once these items are created, they can later be used as building blocks by anyone who has access to the app.

The items can be:

  • Dimensions F
  • Measures F
  • Visualizations F
  • Alternate states F

The benefits of using Master items are:

  • They provide centralized maintenance.
  • They can be reused multiple times in an app, while maintaining them in just one place.
  • When updating them, changes are applied everywhere they are used.
  • They allow the analysis of data over relative time ranges (Calendar measures).
  • They create other selection states, apart from the default state, to analyze the data and compare different subsets together.

Assets panel library

When editing an app, access the Assets panel. The Assets panel is considered a library of items which allow the users to further develop their apps.

Some of the library items are:

  • Fields — A list of data fields available in the app. Users can also reload or refresh the data from this list.
  • Master items — Reusable dimensions, measures, and visualizations created in the app.
  • Charts — The list of standard chart types and objects available in Qlik Sense.
  • Custom objects — A list of different custom extensions and standard extension bundles that you can use in Qlik Sense to create custom visualizations and functionality.

Search and select items in the Assets panel F. From this work space, create, delete, or modify items when the app is not published. When it is published, library items are available for users, but they cannot be modified.

Other app libraries

When performing analysis in the app, you can also generate or create other reusable app items: Bookmarks and Snapshots.

  • Bookmarks save user selections and layout states. You can access them from the Bookmark library.
  • Snapshots store visualizations in a particular point in time and with specific selections applied that can be later used in stories. You can access them from the Snapshot library.

Data islands

A data island is a table that is not connected to any other table in the data structure. This means that when you make selections on the fields that are part of the data island, these selections will not affect the rest of the data model.

To associate the selections in the data island table with the rest of the data model, use Expressions.

Data islands are used to apply additional capabilities without affecting the data model, such as: navigation, dynamic displays, translations, currency exchange rates, etc.

QVD files

A QVD file contains a table of data exported from Qlik Sense or QlikView. QVD is a native Qlik format and can only be written to, and read by, Qlik Sense or QlikView. The file format is optimized for speed when reading data from a script, and it is very compact. Reading data from a QVD file is typically 10–100 times faster than reading from other data sources. A QVD file consists of the following parts:

  • An XML header (in UTF-8 character set) describing the fields in the table, the layout of the subsequent information, and some other metadata
  • Symbol tables in a byte-stuffed format
  • Actual table data in a bit-stuffed format

Use of QVD files

QVD files are used for many purposes. Often, two or more uses are applicable at the same time. At least four uses are listed here:

Increasing load speed

Pulling data from QVDs has a built-in benefit of increasing load speed. In particular, using the Buffer F prefix with non-changing or slowly changing parts of input data in QVD files accelerates script execution.

Decreasing load on database servers

Using QVD files reduces the amount of data fetched from external data sources, lightens the workload on the databases, and decreases network traffic. When several apps share the same data, it is only necessary to load it once from the source database into a QVD file.

Consolidating data from multiple Qlik applications

QVD files are able to consolidate data from multiple Qlik apps. Using the binary script statement, you can only load data from one single app into another. With QVD files, a script can combine data from any number of Qlik Sense or QlikView apps.

Incremental load

QVD files are used to facilitate incremental load processes. Incremental load only loads new or updated records from the databases or data sources and the historic records from the QVD files.

To learn more, search for Loading new and updated records with incremental load F.

QVD files can be read in two modes: standard (fast) and optimized (faster). The optimized mode is only employed when all loaded fields are read without any transformations (formulas acting upon the fields).

QVD Layer

As a best practice, you should always prefer architecture that is based on a QVD layer.

The QVD layer:

  • Uses one or more Qlik apps that load source data, transform it, and store all tables in individual QVD files using the Store command. These apps do not need to keep the data after the QVD files are created. All tables can be deleted after QVD creation.
  • Creates the QVD library that can be updated using scheduled server tasks (data loads).
  • Is not user-facing. A user-facing layer (with a set of apps) is created on top of this layer to load data from the QVD library.
  • Is shared between multiple apps, so they all have the same values and are based on a single version of the truth.

Creating QVD files

A .qvd file can be created in two ways:

  • Using the STORE command - This specifies a table, or part of a table, from the current data model and writes it into a .qvd file. Also, specify a data connection to use as the location for the .qvd file.
  • Using the BUFFER prefix in a LOAD or SELECT statement - Qlik Sense automatically creates a .qvd file in an internal location. This file can be used instead of the original data source when reloading data.

There is no difference between the resulting .qvd files with regard to reading performance. This module explores the first method.

Store statement

The STORE statement in the script:

  • Creates a .qvd or .csv file.
  • Only exports fields from one table in the data model.
  • Exports text values to the .csv file in UTF-8 format, with flexibility to specify a delimiter. However, it does not support BIFF export.

The syntax for the STORE command is as follows:

STORE[ *fieldlist from ] table INTO [ filename ] [ format-spec ];
  • *fieldlist can be an asterisk (*) for all fields or a list of fields separated by a comma that are included in the output file. Field names must match those in the data model, but can also be renamed with an alias in the output file.
  • table is the name of the table from the data model you want to store in the output file.
  • filename is the name of the .qvd or text file: It must include the data connection to the file.
  • format-spec is the format for the file, either txt or qvd. If the format specification is omitted, qvd is assumed.
STORE syntax examples
STORE mytable INTO 'lib://Folder/myfile.qvd'(qvd);
STORE mytable INTO 'lib://Folder/myfile.txt'(txt);
STORE Name, RegNo FROM mytable INTO 'lib://Folder/myfile.qvd';
STORE Name AS Fd1, RegNo AS Fd2 FROM mytable INTO 'lib://Folder/myfile.qvd';

Reading data from QVD files

A QVD file is read by the following methods:

  • Loading a .qvd file as an explicit data source using a LOAD statement in the script, just like any other type of text files.
    Example:
    LOAD * FROM 'lib://Folder/myfile.qvd'(qvd);
  • Automatic loading of buffered .qvd files, using the BUFFER prefix on LOAD or SELECT statements. No explicit statements for reading are required in this case. Qlik Sense determines the extent to which it uses data from the .qvd file instead of from the original data source.
Examples
BUFFER LOAD * FROM MyTable;
BUFFER (stale after 7 days) LOAD * FROM MyTable;
BUFFER (incremental) LOAD * FROM MyTable;
Note: It is a good practice to ensure your QVD loads are optimized for faster readability. If they are not optimized, review the script that creates them. Perhaps, a more suitable design can be found to allow an optimized load.

QVD file script functions

There are a number of script functions that access the data found in the XML header of a .qvd file.

Examples of some of these functions are as follows:

  • QvdCreateTime F (filename) returns the XML header timestamp from a filename file, if any (otherwise, NULL).
  • QvdNoOfRecords F (filename) returns the number of records currently in a filename file.
  • QvdNoOfFields F (filename) returns the number of fields in a .qvd file.
  • QvdFieldName F (filename, fieldno) returns the name of field number fieldno, if it exists in a filename file (otherwise, NULL).
  • QvdTableName F (filename) returns the name of the table contained in a filename file.

Performance considerations

Factors affecting the size of an app

Some of the factors that affect the size of a Qlik Sense app are the following:

Number of rows
  • Examine if it is really necessary to have all the historic data in the app. If yes, analyze the level of detail required.
  • If there are too many detail rows, maybe it is feasible to create a summary table with a higher aggregate level.
Number of columns

You must be selective about which fields you include in a Qlik Sense app.

  • Focus on the larger fact tables first.
  • Do not ignore the dimension tables.
  • Eliminate fields that are not currently being used.
  • All fields may be used in the future, but are not used now.
  • The goal is to have a lean Qlik Sense app.
Number of distinct values / column

Fields with highly distinct values use the most space. Many fields can be eliminated or truncated, such as:

  • System keys
  • Timestamps with minutes / seconds

Others can be transformed by breaking them up into separate components, such as:

  • Phone numbers
  • Postcodes
Data structure

Data structure-related factors that may affect the app size are:

  • Key fields. Use Autonumber where possible.
  • Data model schema. Denormalize the data to reduce the number of tables and joins.
    Note: Qlik Sense stores data internally as distinct lists. Therefore, data redundancy is not a major issue.
  • Large number of variables. Remove them from the script if they are not needed any more.
Number of sheets / visualization objects

Having a large number of sheets and visualization objects affects the size of your app.

Best practices for Qlik Sense app optimization

As the Qlik Sense Data Architect, you build the data model and support your business needs providing a base app to:

  • Support self-service BI users.
  • Support the visualization authors who create reports and stories based on the data.

Data Architects can create complex expressions when working in the app, which can impact performance. When building an app, it is important to be aware of the user's requirements. As a result, you can create:

  • Specific logic in the script to simplify complex expressions.
  • Master items (Dimensions and Measures) using optimized calculations.

App design

You can improve the design of your app and optimize its performance by doing the following:

  • Reduce the number of visualizations per sheet. It is better to have more sheets with fewer objects in each than a few cluttered sheets.
  • Avoid complex expressions and calculated dimensions. If possible, in the Data load editor, pre-calculate parts or the whole calculation.
  • Analyze if all the fields and the tables containing them are required in the data model. Tables and Pivot tables, combining several fields from different tables, can impact the performance of the application.
  • Use set analysis, if required, but do not use it by default when creating expressions. Remember that users can perform selections in the app to dynamically analyze the data.
  • Create a dashboard or overview sheet with aggregated tables containing a large volume of data.

Resource-intensive expressions

There are some functions and expressions that also affect the performance of your app. To improve the response time of your app, you can do the following:

  • If (Condition(Text),...)
    Conditions comparing text strings are resource-intensive. Analyze if the conditions can be changed using numeric fields instead.
  • Sum (If (Condition, 'FieldName'...))
    Here, the IF statement is evaluated at row level. The sum calculation is performed at Dimension level. The problem can be treated either by performing the logic in the script and aggregating in the table, or by performing the whole operation in the script.
  • If (Condition, Sum('FieldName')...)
    This option emphasizes the difference in comparison to the previous expression. This aggregation is completely contextual and less resource-intensive than the previous one.
  • If (Condition1, Sum('FieldName'),
          If (Condition2, Sum('FieldName')...)
    The logic of nested If…Then…Else is conceptually easy, but can often become troublesome to administer. You can quickly create examples with many nesting levels. However, this expression is memory and CPU-intensive. You can often replace the conditions by transforming them. A typical example is aggregating quantity * price, where price is variable. This can be handled by using an extended IntervalMatch function.

Sort text

Qlik Sense automatically evaluates if a field is to be treated as numeric, text, or general. Fields evaluated as text are sorted as text, which is the slowest sort operation. You can manually change this to sort by load order.

Dynamic captions and text objects

Enter expressions in the properties for visualization objects wherever you see the 3 icon. Expressions are recalculated with every user selection, but only for the objects on the current sheet.

Adding aggregate fields into your script

Add aggregate fields into your script, instructing your document to sum up these values. Complex statements can be placed in the Data load editor to simplify the visualization objects.

Example:

The IF statement is calculated in the script to create a field as either 1 or 0, enabling you to Sum or Count the field.

The script appears as follows:

IF(ACTIVE='Y',1,0) AS ActiveFlag
IF(ACTIVE='Y',sales_amount,0) AS ActiveSales

Sum and Count in the measures appear as:

COUNT(ActiveFlag)

SUM(ActiveSales)

Forcing a 0 takes out all of the unnecessary and unwanted values when you apply your Sum.

HidePrefix and HideSuffix

It is good practice to hide key fields to prevent them from being used in expressions by the Business User, which can lead to incorrect results. All field names beginning or ending with a specific text string can be hidden in the same manner as system fields. The user defined variables, HidePrefix and HideSuffix, can be used to hide fields.

SET HidePrefix ='%';

If you use this statement, the field names beginning with a percent symbol are not visible as fields to the Business Users.

Name key fields with a recognizable string, thereby hiding them when created in the load script. Optionally, rename the fields that you want to hide with a series of Rename statements at the end of the load script.

RENAME FIELD OfficeID TO %OfficeKey;
RENAME FIELD CustomerID TO %CustomerKey;
