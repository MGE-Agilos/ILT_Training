Data Modeling for Qlik Sense

Activity 4: Join Categories with Products

Summary:

  • Evaluate the association between the Categories and Products tables.
  • Join the tables to further fold the data model in from a snowflake to a star schema.

Module: Combining tables

Qlik Sense app: ABC Sales Data

Challenge

In the ABC Sales Data app, join the two tables, Categories and Products, by completing the tasks below:

  • In your data model, analyze the Categories and Products tables and review the following:
    • Number of rows or records
    • Number of fields or columns
    • Field(s) linking the tables
    • Density of the CategoryID key field
    • Subset ratio of the CategoryID key field
    • Tables linked with other tables
    • Key fields in each table
    • Field data types: Are there any numeric fields that can be affected in case a duplication happens?
  • Create a new app sheet, Categories and Products, and add two tables:
    • The first table contains two columns: ProductID as the dimension and Count(CategoryID) as the measure.
    • The second table contains CategoryID as the dimension and Count(ProductID) as the measure.
  • In the Products script section, locate the Categories table.
    • Comment out the name of the table.
    • Add the JOIN prefix in front of the LOAD statement. Make sure that you specify the table you want to join. In this case, choose Products by adding it inside the brackets.
  • Load your data and check the results in both the Categories and Products sheet and your data model.

Hints

  • Joining tables is a technique that contributes to the denormalization of the data model.
  • The type of JOIN needs to be defined first, before being applied in the script.

 

Step-by-step guide
  1. Open the Data model viewer for the ABC Sales Data app.
  2. Analyze the Categories and Products tables by reviewing the following:
    • Number of rows or records. (8 in Categories and 78 in Products)
    • Number of fields or columns. (3 in Categories and 11 in Products)
    • Which field or fields are linking the tables? (CategoryID)
    • Density of the CategoryID key field. (100% both)
    • Subset ratio of the CategoryID key field. (100% both)
    • Are any of these tables linked with other tables? (Yes, Products is linked with other tables)
    • What are the key fields in each table? (CategoryID, SupplierID, ProductID)
    • Preview the data and review the data types of the fields: Are there any numeric fields that can be affected in case a duplication happens? ( Yes, in Products: ItemCost, ItemPrice, UnitCost, QuantityperUnit, CataloguePrice, UnitsInStock, UnitsOnOrder)
    3. Note: In a 1 to 1 relationship, one record from one of the tables corresponds to only one record of the other table. The Categories table has fewer records than the Products table. In this case, the relationship is one record corresponding to several records (1 to N).
  3. Create a new app sheet to analyze the table relationship further. Name the sheet Categories and Products.
  4. Insert a Table onto the left half of the new sheet, with ProductID as the dimension and Count(CategoryID) as the measure.
  5. Insert a second Table onto the right half of the sheet, with CategoryID as the dimension and Count(ProductID) as the measure.
  6. Make sure both tables are sorted by measure.
  7. Finish editing the sheet and explain the relationship between Categories and Products (1 to N). Consider the technique (JOIN or MAPPING) that works best.
    The relationship between Categories and Products is 1 to N. There is no risk of duplicating the records in the Products table if we combine both tables using a JOIN. JOIN is best in this case because:
    • A JOIN will produce a table with 78 records and 13 fields, with no NULL values.
    • A Mapping Table only accepts two fields and the Categories table has three fields. If you want to use this technique, you need to create and maintain two mapping tables.
  8. In the Data load editor, locate the Products section.
  9. The Categories table contains a field called Description. After the JOIN, this field will be part of the Products table. To avoid confusion, it is recommended that you alias the field to CategoryDescription. This specifies that the Description field refers to Categories.
    ...
    Description AS CategoryDescription
    ...
  10. Comment out the Categories table name and use the JOIN prefix. Note, if the type of JOIN is not specified, Outer Join will be the result:
    //Categories:
    JOIN(Products)
    LOAD CategoryID,
    ...
  11. Load your data. Check the results in the Categories and Products sheet and in the Data model viewer. Notice that the fields of the Categories table have now been added to the Products table.
    The number of tables in your data model has been reduced by one. You are one step closer to a star schema.
    Note: The above steps explain in detail why it is acceptable to use Join to form two tables into one. If you do not specify the Join type, the Join prefix will combine the two tables using Outer Join.
    It is a best practice to explicitly define the Join type you want to use. In this case, although using Join will transform the data model in the same way as a Left Join, it is recommended to use Left Join.
