Data Modeling for Qlik Sense

Activity 7: Employees concatenation

Summary:

  • Append two new employees to the Employees table.
  • Investigate the behavior of forced and automatic concatenation of table data.

Module: Combining tables

Qlik Sense app: ABC Sales Data

Challenge

In the ABC Sales Data app, load a new Excel file to add two new employees in the data model. To do this, complete the following tasks:

  • In your data model, verify that the Employees table contains 62 rows.
  • In the script, modify the FROM clause in the Employees table and add a wildcard (*) to the end of the filename.
  • Load your data and verify that the Employees table now has 64 rows.
  • Revert the FROM clause back to its original value and create a copy of the Employees LOAD statement right underneath.
  • If required, upload the employees_new.xlsx file to your Personal space in Qlik Cloud.
  • Edit the new FROM clause, in the second LOAD statement, using the employees_new.xslx filename.
  • Since the two tables do not have exactly the same structure, use the CONCATENATE prefix to force concatenation.
  • Add the new field EmploymentType in the new table with the value Permanent. Now the two tables have identical structure.
  • Comment out the CONCATENATE prefix and load your data. Verify that the Employees table still has 64 rows.

Hints

  • The structure of the employees_new.xlsx is similar to the employees.xlsx.
  • Both files are stored in the same location and their names start with employees. Thus, a wildcard can be added in the script to load them.
  • When wildcards are used as parts of the filenames in the FROM statements, certain business rules need to apply to avoid script errors and ensure consistency:
    • All the files need to have identical structure.
    • The Business User should never modify the field names or remove columns in any of the source files.
    • If new columns are added in the source tables, and their field names are not included in the LOAD statement, they will be ignored.

 

Step-by-step guide
  1. Locate the Employees table in your data model and preview its data. Verify that there are 62 rows in this table.

Using a wildcard to load files.

  1. Use the Select data option on the DataFiles connection, upload the employees_new.xlsx file from the student collateral, and cancel the Select file dialog when the upload is complete.
    2. Qlik Sense on Windows: The employees_new.xlsx file already exists in the ExcelFiles connection.
  2. In the Employees script section of the Data load editor, locate the Employees table and add an asterisk to the end of the filename, as follows:
    FROM [lib://<ConnectionName>/employees*.xlsx]
  3. Load your data. In the data model, verify the number of rows in the Employees table. There are 64 rows in this table; the 2 new rows have been added automatically.
    This is possible because both files are stored in the same location and have identical structure. The wildcard can be used to load several tables with the same structure at once. Qlik Sense concatenates the records of all the tables with the same structure in one single table automatically.

Forced concatenation

  1. Open and review the new employees file, employees_new.xlsx. Note that there is a new column in the employees_new.xlsx file called EmploymentType. This field is not in the employees.xlsx file.
    You can load this field into the new Employees table. But, this makes the new table structure different than the Employees table. These tables will need to be concatenated.
  2. In the script, locate the Employees table again and revert the FROM clause back to its original value:
    FROM [lib://<ConnectionName>/employees.xlsx]
  3. Copy the entire LOAD statement for the Employees table and paste it immediately after that statement. Do not include the table label in your selection.
  4. Edit the FROM clause in the new LOAD statement to read as follows:
    FROM [lib://<ConnectionName>/employees_new.xlsx]
  5. Add the new field EmploymentType in the LOAD statement of this new table. Right now, the tables do not have the same structure and will not be concatenated together unless you force the concatenation.
  6. Between the two tables, just above LOAD , add the following script:
    CONCATENATE (Employees)
    The script for the two tables resembles the following:
    Employees: LOAD EmpID AS EmployeeID, "First Name" & ' ' & "Last Name" AS Name, DateOfBirth, StartDate, EndDate, OfficeID, JobTitle FROM [lib://<ConnectionName>/employees.xlsx] (ooxml, embedded labels, table is Employees);

    CONCATENATE (Employees)

    LOAD EmpID AS EmployeeID, "First Name" & ' ' & "Last Name" AS Name, DateOfBirth, StartDate, EndDate, OfficeID, JobTitle, EmploymentType FROM [lib://<ConnectionName>/employees_new.xlsx] (ooxml, embedded labels, table is Employees);
  7. Load your data. In the data model, verify that the Employees table still has 64 rows. A new field called EmploymentType has been created. This field contains Null values for all the employees from the Employees.xlsx.
  8. On the Employees sheet, search for the two new employees in the Name filter to confirm their addition. Remember that the search control is case-sensitive.

Auto concatenation

  1. Set a default value for the new field, EmploymentType, in the Employees table for all the records with a NULL value: In the Data load editor, open the Employees script section and add this new line in the LOAD statement for the Employees table, just after JobTitle (remember to put a comma after the JobTitle field):
    'Permanent' AS EmploymentType
  2. Because you added the EmploymentType to the first employee table, both tables have the same structure again and you can comment out the CONCATENATE statement:
    // CONCATENATE (Employees)
  3. Load your data and preview the Employees table in your data model. Verify that it still has 64 rows and the Density of the EmploymentType is 100%.
  4. Within Analyze, open the Employees sheet and enable the edit mode.
  5. Add a new Filter pane onto the sheet with the EmploymentType dimension and observe its values.
