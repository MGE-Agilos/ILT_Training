Adding calculated fields to a table

In addition to loading existing fields of a table, Qlik Sense allows you to create new fields as well.

The new fields:

  • Can be created during the execution of the data load script to provide the data model with additional data values.
  • Are based on existing fields in the data model or based on transformations using Qlik Sense functions and calculations.
  • Are used to ensure that the front-end visualizations in Qlik Sense apps meet the business requirements.
  • Are used to perform any complex calculations in the Data load editor rather than calculating on-demand in the front-end.
  • Ensure real-time updates of visualizations on the front-end as the Business Users make selections, investigate the results, and perform their analysis.

Script functions in Qlik Sense

There is a large collection of Qlik Sense functions used for front-end calculations and for maneuvering data during the data loading process.

Many of these functions can be used in the same way for both data load scripts and chart expressions. However, there are a number of exceptions:

  • Some functions can only be used in data load scripts (script functions).
  • Some functions can only be used in chart expressions (chart functions).
  • Some functions can be used in both data load scripts and chart expressions, but with differences in parameters and application.
Function example - Year()

To return a value representing the year component of a date, use the YEAR function and provide the field holding the date within the parentheses. This function is both a script and chart function.

YEAR (OrderDate)

Note: While field names in Qlik Sense are case-sensitive, function names are not, which means YEAR or year will not have any deviations in the results.

The syntax in the preceding example works. However, after executing a data load, the name of the new field you have created is YEAR(OrderDate).

To provide an alternative name for this field in the model, you must alias the field by using the AS keyword, as follows:

YEAR (OrderDate) AS Year

When text is typed into the script, the Data load editor offers a tooltip that lists Qlik functions, arguments, and qualifiers based on what you entered.

Limiting and re-using data

The Where clause

The WHERE clause uses a condition to determine the records for loading. Only the records that meet the condition are loaded from the source.

To help improve performance, it is a best practice to load only the records required for the analysis.

The Where Exists clause

This clause determines the inclusion of a record based on the values that exist in the data.

Exists(field [ , expression ] )

  • Field is a column name. This field must exist in the data that has been loaded.
  • Expression is a field or calculation, resulting in a value. If omitted, the values in the specified field are assumed.
Note: Where Exists is very commonly used for Incremental load scenarios (see Other related topics below).

Example:

Where Exists(IDnr_Order, IDnr)

The Exists function returns -1 (true) if the value of the field IDnr in the current record already exists in any previously read record in the IDnr_Order field.

Resident load

Use a LOAD...RESIDENT when:

  • Creating a new table based on another table previously loaded into the script.
  • Performing aggregations or group-by calculations in an existing table.
  • Creating complex calculations that require several steps and pre-calculated formulas.

Creating Composite keys

A composite key can be created by concatenating two or more fields into one, using ampersands (&) as follows:

OrderID & LineNo AS OrderLineKey

In this case, both OrderID and LineNo are numbers and combining them results in a new numerical value. This sometimes leads to unexpected behavior.

Examine the following scenario:

OrderID LineNo OrderLineKey
1 11 111
11 1 111
NULL() 123 123
123 NULL() 123

When rows from two different orders are associated using this technique, as shown in the first two rows in the table above, they get the same OrderLineKey, even though the OrderID and LineNo values are different. This can lead to data integrity issues and must be avoided.

A solution is to place a separator between the two numbers. Use a character that is not contained in the values of the keys you are combining.

Here are some examples:

OrderID & '-' & LineNo AS OrderLineKey
OrderID & '%' & LineNo AS OrderLineKey
OrderID & '|' & LineNo AS OrderLineKey

Using some of these examples, the results would look like this:

OrderID LineNo OrderLineKey with | separator OrderLineKey with - separator
1 11 1|11 1-11
11 1 11|1 11-1
NULL() 123 |123 -123
123 NULL() 123| 123-

Adding a separator converts the values in the OrderLineKey field to a text value instead of a numerical value. Avoid using currency or mathematical symbols like minus (-) or plus (+) as separators. They can be interpreted as numeric/currency signs. See the example highlighted in red above.

Adding the separator:

  • Prevents the concatenated text from creating the same key from different values, keeping each key unique.
  • Makes the key easier to read and understand during development because you can see the individual values that make up the key.

Whenever a composite key is used to resolve a synthetic key, keep the fields that make up the key in one table, if they are useful for reporting.

Overview of synthetic keys

Qlik Sense creates synthetic keys when two or more loaded tables have more than one field with the same name.

The synthetic keys are:

  • Generated automatically (for example, using IntervalMatch F).
  • Anonymous fields that represent all occurring combinations of the composite key.
  • Generally resource-heavy and may slow down calculations and, in extreme cases, overload a document.
  • Hard to understand and maintain.
  • Responsible for circular references in some cases.

Synthetic keys need to be eliminated, if possible. An overall analysis of the intended table structure, by the Data Architect, is recommended.

Synthetic key typical solutions involve:

  • Renaming fields.
  • Forming your own composite keys, typically using string concatenation inside an AutoNumber F script function.
  • Removing unnecessary fields, ensuring that only the necessary fields remain and connect.
  • Combining the tables affected by the synthetic key or forming your own Link table with all the common keys.

AutoNumber

When concatenating long composite keys with several key fields, you can use the AutoNumber function to get a number instead of text for the key field. This improves performance when working with large tables.

This function gives every unique instance of OrderLineKey a numeric representation.

Example:

AutoNumber(OrderID &'|'& LineNo) AS OrderLineKey

OrderID LineNo

without AutoNumber
(using | separator)

 with AutoNumber
668 110 668|110 1
668 111 668|111 2
669 112 669|112 3
668 111 668|111 2

AutoNumber also accepts a second, optional parameter to create multiple counter instances if the function is used for several composite keys.

Example:

AutoNumber(OrderID &'|'& LineNo,'OrderLine')

The values are generated in each script execution, according to the order the table is read and the values already loaded.

Note: You can only connect the autonumberhash keys that have been generated in the same data load, as the integer is generated according to the order the table is read. If you need to use keys that are persistent between data loads, independent of source data sorting, you should use the hash128, hash160 or hash256 functions.

Master calendar

Creating a table to keep the fields related with the date dimension is often a practice used by Data Architects. A master calendar table in the data model:

  • Keeps the date fields outside the central measures tables.
  • Fills in potential date gaps that the fact tables might contain. It contains all the dates, not only dates with related data.
  • Can combine several dates in one single calendar.

When displaying data trends, it is important to also display the periods with no activity. A master calendar is usually generated based on a start and end date from one or more fact tables.

Qlik Sense has the ability to extract date-related information from the table fields that contain dates. It automatically recognizes the fields tagged as $date and uses them to create a set of virtual calendar fields that can be added into calculated expressions. This is achieved by using the Derive F statement.

The Data manager must be used to load the data (or the data must be synchronized after it is loaded, by using the Synchronized scripted tables F option, from within the Data manager).

The autoCalendar table and the derived fields are more than enough for Business Analysts. However, the calendar fields are virtual and cannot be seen in the Data model viewer. This means that they cannot be used as part of the data model transformation.

The creation process of a MasterCalendar table is as follows:

  1. Use a LOAD...AUTOGENERATE statement to create a new temporary table with the start and end dates of your fact table. The FieldValueCount is used to return the number of distinct values of the field. This defines the length of the new table. In the memory, the script initially creates a table with the same number of rows as the symbol table and the two fields that contain all the distinct date values. Eventually, and after aggregating these values using the Min and Max functions, this table ends up with two columns and one single row. The most efficient way to retrieve the minimum and maximum dates for this table, is to use the FieldValue function.
    Note: The FieldValue and FieldValueCount functions access the data directly in the Symbol tables that Qlik Sense creates in memory, where only distinct values are stored when the table is loaded.
    This means that for a table with 100M rows that contain transactions for a specific month, Min and Max will process all the 100M rows to find the smallest and largest values. In the case of the FieldValue function, there are only 31 distinct values that will be processed in the symbol table.
    For more information on symbol tables and how Qlik Sense stores data in memory, review the post on Symbol Tables and Bit-Stuffed Pointers (Qlik community).
  2. A Resident Load statement is used to transform the existing table. In this new version of the table, the number of rows is the same as all the days in between the start and end dates. The new table begins from the minimum date and ends in the maximum date, with all the dates in between (no gaps). A While statement is used here to control the iteration.
  3. A second Resident Load is used to transform the table created in the previous step, to the main MasterCalendar table. Use date/time functions and logic to create the date fields needed, such as Year, Month, and Day.
    Note: A preceding Load could be used instead for the above transformation. This is a concept very commonly used to further transform the tables and the data model. It allows you to load a table in one pass and add several transformation steps.
    For more info, review the post on Preceding Load (by Henric Cronström).

Date and time in Qlik Sense

Qlik Sense stores each date, time, and timestamp found in data as a datetime serial number. Dates and times can be added and subtracted, and intervals compared.

The date time for 01/01/2014 06:00:00 is represented as 41640.25, where:

  • The integer part, 41640, represents the date and is the number of days elapsed since December 31, 1899 (date 1).
  • The decimal part represents the time stored as a value between 0 and 1. This number is the fraction of elapsed time from a full day. Therefore, 06:00:00 is six hours elapsed from a full day, or 0.25. The value 0.0 corresponds to 00:00:00 and 0.99999 corresponds to 23:59:59.

Example:

Value Format Text displayed
44312 'M/D/YYYY' 4/26/2021
44312.65208333 'YYYY-MM-DD hh:mm:ss' 2021-04-26 15:39:00

When you load data into Qlik Sense, fields containing numbers with a defined data type in a database are handled according to their respective formats.

The following functions can be used to define input and output formats of dates:

  • Date(), Time() — Formatting functions that define the presented data in Qlik Sense.
  • Date#(), Time#() — Number interpretation function that define the data format of the date being loaded into Qlik Sense.

Flags

A MasterCalendar table usually contains Flags. These Flags are fields that:

  • Are created using Qlik Sense functions or IF statements.
  • Mark-up every date in the calendar using 1 or 0, matching the appropriate condition. Examples include the current year (CYTDFlag) and the current period of the last year (LYTDFlag).
  • Are useful when comparing results, such as current year vs. last year.
  • Can be used in different ways, such as multiplying the value in a field with the value of the flag. For example:
    SUM(LineSalesAmount * CYTDFlag)
    If CYTDFlag is set to 0, the result is 0. But, if the flag is set to 1, the result is the LineSalesAmount value.
  • Can also be used together with Set Analysis expressions.

The InYearToDate function

One of the functions used to calculate the CYTDFlag and LYTDFlag is the InYearToDate function. The function syntax is as follows:
InYearToDate(date,basedate,shift [,first_month_of_=1])

The InYearToDate function:

  • Returns true if a date lies inside the part of the year containing basedate up to and including the last millisecond of basedate. The year can be offset by shift. Shift is an integer where the value 0 indicates the year that contains basedate. Negative values in shift indicate preceding years. Positive values indicate succeeding years.
    Example

    If the current date is Mar 31, 2020:

    OrderDateCYTDFlagLYTDFlag
    Mar 30, 201901
    Mar 31, 201901
    Apr 1, 201900
    Mar 30, 202010
    Mar 31, 202010

  • Is helpful when you want to compare the results of the current year with the previous year. Because you only have data up until today for the current year, you may not want to compare against the entire previous year, but instead with the same time period in a previous year.
Time-aware charts
Time-aware charts are visualizations that use a continuous scale to provide a complete and accurate view of time-based data, even if there are gaps in your data set. Continuous scale is only supported for Line charts.
You can find more information on Time-aware charts F in the Help site.
