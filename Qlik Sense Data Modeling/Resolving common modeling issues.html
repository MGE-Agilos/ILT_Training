Synthetic keys

  • Qlik Sense creates a synthetic key when two or more tables have more than one common field name but not exactly the same structure.
  • A synthetic table contains a synthetic key which lists the common field names. Qlik Sense creates synthetic keys when two or more data tables have two or more fields in common.
  • A synthetic key will be denoted as $Syn1. If there is more than one synthetic key, the suffixed number value will be displayed accordingly (example: $Syn2, $Syn3, etc...).

Note: Field names in Qlik Sense are case-sensitive. Links between fields do not occur if the characters do not match by name and case.

Other related topics:

  • Synthetic keys F
  • Synthetic Keys (by Henric Cronström)

Why do synthetic keys need to be avoided?

Synthetic keys and tables:

  • Become another key and table in the data model, which the associative engine has to take into consideration when users make selections.
  • Might generate wrong results if the synthetic key is not a valid association. Even though there are fields with same names, their meanings might differ.
  • Are usually caused by an unintentional association between tables. Qlik Sense displays a warning when a synthetic key is created.
  • Are resource heavy and can slow down calculations and overload a document.
  • Make a document harder to understand and maintain. The synthetic keys are not available for preview or usage in the user interface.
  • Are impossible to control what they contain as there is no script in the Data load editor that creates this table. This also makes it impossible to transform its fields.
  • Are responsible for circular references in some cases.

  • Should We Stop Worrying and Love the Synthetic Key? (Qlik Community)

Resolve a synthetic key issue

The way to remove a synthetic key is to modify the load script in a way that the model will no longer have two or more fields with exactly the same name on multiple tables. For this it is important to know the data structure and the data sources to create the correct solution.

  • Prune the non-required fields if there are fields that can be removed from one or more tables.
  • Alias or rename one or more fields if they do not represent the same concept.
  • Create a composite key (Covered in other modules) in the tables if all fields are required and their names cannot be changed because they are part of a compound key.
  • Combine tables together if all fields are required and their names cannot be changed.
  • Create a link table, which is similar to the synthetic table, but is maintained and controlled by the Data Architect.

Counting table records

Key fields are fields that are common to one or more tables. Values for each key field are stored in one place in memory, even if those values came from different tables. Because of this:

  • It is not possible to show the Frequency for key fields.
  • A key field in one table may have fewer distinct values than the same field in another table. This can be analyzed by checking the Subset ratio.
  • Count key fields can return misleading numbers as the count is performed over all the possible values, in all the tables that contain the field.

Resolving issues associated with counting field values

Loading a field into a table multiple times

To calculate the Frequency of a key field in a table, load the key field twice in the table and rename one instance, so it is no longer a key field. Use the new field to calculate the frequency.

Example:
LOAD …, OrderID,OrderID AS OrderIDCount …FROM Orders;

With a new field name within the data model, you can perform a count for all occurrences, or a distinct count to find the unique occurrences.

Using a record counter on key fields

Use a record counter in the table where you want to make the calculation. A record counter can simply be the value 1, set for each row in the table. For instance, if you want to count the number of orders, create a record counter in the Orders table and use the Sum() of the record counter in charts.

Example:
LOAD …, OrderID,1 AS OrderRecordCounter …FROM Orders;

This method is more efficient, but in some cases using a record counter is not possible. For instance, review these two scenarios:

  • Scenario 1: Find the number of customers. Create a record counter in the Customers table, as it is the master table for customers. Then, perform a simple Sum() of the field.
  • Scenario 2: Find the number of customers with orders. Identify the customers with orders by duplicating the CustomerID field in the Orders table. Then, apply a Count(Distinct) of this new field.
Note: Be careful when creating both Count() fields and record counters: Create the new field in the correct table.

Circular references

When you work with more complicated data models that contain many tables, it is possible to create a situation where the interpretation of the data is uncertain.

Qlik Sense is equipped to interpret complex models. However, there are some limitations. It is important to be aware of these limitations and know how to solve the problem of loops when they arise.

The following example consists of a simple data structure with three tables:

  • There is a circular reference between the three tables.
  • Each of the tables has a field that links to another table, creating a loop around the tables.
  • There is more than one path between tables, which can lead to an ambiguous interpretation of the data.

Circular references are quite common, caused by poor database design or by replicating the structure from the source data.

Occasionally, a field (or a table) may have several roles. For example, a company may be both a supplier and a customer. The field (or table) must then be loaded into Qlik Sense twice using different names.

Other related topics:

  • Understanding and solving circular references F
  • Circular References (by Henric Cronström)

Loosely coupled tables

Qlik Sense solves the problem of circular references by defining a loosely coupled table. This means that it will temporarily disconnect the table from the rest of the data model. If Qlik Sense finds a loop while performing a data load, an error is displayed in the Data load progress dialog, and one or more tables are set to loosely coupled in the data model.

Qlik Sense attempts to make the longest table (most rows) loosely coupled. This is often a transaction table. If you wish to deviate from the Qlik Sense default, you can define the table to be loosely coupled using a Loosen Table statement in the script.

In addition, you can determine which tables in your structure are set to loosely coupled by using the Data model viewer. Loosely coupled tables show red dotted lines as their connectors.

Basic data transformation

The LOAD statement is defined in the Data load editor to load and transform data in the application.

There are different types of loads based on the source of the data.

  • Use Load...From for typed ASCII files, .xlsx, .xls, .csv, .txt, .qvd, .html, and .xml.
  • Use Load...Resident for data defined from a previously loaded table.
  • Use a preceding load (LOAD...;) for fields that are the result of a SELECT or another LOAD statement. The subsequent SELECT or LOAD must immediately follow the preceding LOAD statement.
  • Use Load...Inline [ ... ] for data defined directly in the script.
  • Use Load...Autogenerate for data auto-generated in the script.
Load statement syntax example 
       LOAD 
       [ distinct ] *fieldlist
	[( from file [ format-spec ] | from_field
	field as source [ format-spec ]
	inline data [ format-spec ] | resident table-label | autogenerate size )]
	[ where criterion | while criterion ]
	[ group by groupbyfieldlist ]
	[ order by orderbyfieldlist ]
Note: An advantage of the LOAD statement over the SELECT statement is the ability to use all of the Qlik Sense script functions.

The following are examples of LOAD statements:

Example 1: Load...From and Preceding Load in Load statement
LOAD Year, "First Name" & ' ' & "Last Name" AS Name;
LOAD * FROM [lib://textfiles/data.txt] (ansi, txt, delimiter is '\t', embedded labels);
Example 2: Preceding Load in Select statement and Load...Resident
ABC_Tab1:
LOAD A AS [User Name], C+B AS Sales, Year(D) AS Year;
SELECT A, B, C, D FROM [lib://dataconnection/Table1];

[New Table]:
LOAD Year, Sum(Sales) AS [Year Sales]
RESIDENT ABC_Tab1 GROUP BY Year;
Example 3: Load Inline and Load...Autogenerate
LOAD * INLINE
[VisitID, Visitor type, Estimated Time (hours)
0, Regular visitor, 0.45
1, Occasional visitor, 2
2, Permanent worker, 7.5];

LOAD RECNO() AS ID, RAND() AS Num
AUTOGENERATE(100);
Note: Field names and table names must be enclosed by double quotes or square brackets if they contain spaces or special characters, with the exception of the fields defined inside the Inline statement (refer to Example 3 above).

The LOAD statement accepts prefixes such as Sample and First to load just some records, and others like Hierarchy, HierarchyBelongsTo, CrossTable, and Generic to transform the table structure.

Renaming a field

It is possible to rename one or more fields in the script in multiple ways:

As statement

Use AS in the LOAD statement to rename a field from one table. When you use Select data, you can click on any field in the Fields area and type a new name. The generated script includes AS.

Example:
LOAD Capital AS [Capital city],
Pop AS Population
FROM [lib://textfiles/Country.csv] (ansi, txt, delimiter is ',' , embedded labels);
Alias statement

Use ALIAS to rename all the occurrences of one or more fields in the subsequent tables temporarily. Aliases are often used to make field names more readable or meaningful. This is the syntax:

ALIAS fieldname AS aliasname {, fieldname AS aliasname}
Example:
ALIAS Mon AS Month, Cname AS [Customer Name];

Search for Alias F in the Help site.

Rename Field statement

Use RENAME FIELD to rename one or more existing fields after they have been loaded. This statement can optionally use a mapping table, which stores the oldname to newname conversion data. The syntax is:

RENAME FIELD[S] (using mapname | oldname to newname {, oldname to newname} )
  • mapname represents the name of a previously loaded mapping table containing one or more pairs of old and new field names.
  • oldname represents the old field name.
  • newname represents the new field name.
Note: Both Rename Field and Rename Fields are accepted syntax.

Examples:

RENAME FIELD ABCSales TO Sales;

FieldMap:
MAPPING LOAD oldnames, newnames FROM TDictionary;
RENAME FIELDS using FieldMap;

Search for Rename F in the Help site.

Qualify and Unqualify statements

The QUALIFY statement renames fields in the tables that follow it, in the form of tablename.fieldname. Tablename is the label of the current table. If no label exists, it considers the name appearing after the FROM in LOAD and SELECT statements.

You can stop renaming fields by using the UNQUALIFY statement.

The syntax for each is:

QUALIFY *fieldlist;
UNQUALIFY *fieldlist;

where fieldlist is one or more field names separated by commas.

All fields in all LOAD statements following a QUALIFY statement are qualified until an UNQUALIFY statement is used to switch off qualification.

 Example 1
QUALIFY *;

T1:
LOAD A,B FROM [lib://dataconnection/T1];

T2:
LOAD A,B,C FROM [lib://dataconnection/T2];

UNQUALIFY *;

This results in fields named T1.A, T1.B, T2.A, T2.B, and T2.C. The fields in the tables following the UNQUALIFY statement are not renamed.

 Example 2
QUALIFY A,B;

T1:
LOAD A,B FROM [lib://dataconnection/T1];

UNQUALIFY A;

T2:
LOAD A,B,C FROM [lib://dataconnection/T2];

This results in fields named T1.A, T1.B, A, T2.B, and T2.C.

Note: The qualified fields can be viewed in the Data model viewer.

Search for Qualify F and Unqualify F in the Help site.
