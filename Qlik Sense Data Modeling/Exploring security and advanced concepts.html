Big Data with Qlik Sense

There are times when loading an organization's data in-memory for analysis is extremely difficult due to the size of the data sets.

It can be very difficult to fit very large amounts of data into memory. In addition, transferring large amounts of data across the network from data storage can take too much time.

Qlik Sense uses the following methodologies for handling Big Data:

  • On-Demand App Generation (ODAG)
  • Direct Discovery
  • Document Chaining
  • In-Memory
  • Segmentation

There is not a one solution fits all approach for handling Big Data. A combination of the above needs to be used in many cases, depending on the complexity, data volume, and use case.

Note: This module focuses on the first two methodologies of Big Data handling.

On-Demand App Generation (ODAG)

What is ODAG?

  • A method of handling Big Data or near real-time data with Qlik (both Qlik Sense and QlikView)
  • A shopping cart approach to analytics
  • A governed method for user-driven data loads

  • A conversion of Big Data into small data analytics by creating on-demand data slices.

To use ODAG you need to:

  1. Enable the On-demand app service in the product management settings page.
  2. Develop a Template app (Analysis app) with the data model and the UI (sheets / visualizations).
  3. Add the ODAG script to the Analysis app to capture the filter from the Selection app.
  4. Develop the Selection app.
  5. Configure the App navigation link.
  6. Add the App navigation link object to the sheet.
  7. Make selections and click the Generate new app button to create the new app for more detailed analysis.

How does ODAG work?

  • Two apps need to exist in order for ODAG to work:
    • A Selection app populated with dimensional data and high level aggregated tables
    • An Analysis app (Template App) that contains the detailed information
  • The user selects a slice of data in the Selection app.
  • The user requests the Analysis app to be built.
  • The Analysis app is created and posted in a predefined space or stream (either published or unpublished).

Dynamic views

In Qlik Sense Enterprise, Dynamic Views are enabled by your administrator.

Dynamic views are accessible from the sheet editor > Assets panel. Dynamic views allow app sheets to contain charts that are loaded from data sources on-demand.

With the Dynamic Views option turned on, you can refresh charts from within your analytic tool environment, without having to reload the app.

To enable Dynamic views, do the following:

  1. In the Management Console, go to the Settings page.
  2. Under the On-demand data section, switch on the On-demand app generation button and Enable dynamic views button.

Managing security with Section Access

Qlik Sense uses the information in SECTION ACCESS to:

  • Handle security. The security is built into the script editor in the app. If the app is exported from Qlik Sense, it is still protected.
  • Authorize users to open an app and define the user access levels.
  • Dynamically reduce the data so that users only see the data they are allowed to see. In this way, an app can be used to internally hold the data for a number of users or user groups.

When an app is published or in a shared space, the security is analyzed to only allow specific users to access the app. If the app also has SECTION ACCESS, this applies after, as a second level of security.

Access control

Access control is managed through one or more security tables loaded into the app through the load script.

The access tables are:

  • Stored internally as part of the same database.
  • Not visible in the Data model viewer.
  • Defined in the script with a SECTION ACCESS statement. The main body of the script that creates the data model is called SECTION APPLICATION.

The SECTION ACCESS statement defines the part of the script where the security tables are located, up until the next SECTION... statement is reached.

SECTION ACCESS;

The SECTION APPLICATION part specifies the data tables in the app, until the next SECTION... statement is reached.

SECTION APPLICATION;

If no section is specified in the script, SECTION APPLICATION is assumed.

Section Access tables and system fields

The SECTION ACCESS tables:

  • Are tables used for authentication or authorization. They are defined in the SECTION ACCESS in the script, before the SECTION APPLICATION.
  • Follow the same associative logic as tables in the Qlik Sense data model. Therefore, security fields can be put in different tables.
  • Can be managed using text files, databases, or INLINE clauses in the same way that data is usually handled by Qlik Sense.
  • Contain the following system fields, that must be in UPPERCASE (including field names and values), except for the OMIT field values. Any field values defined in these tables are converted to uppercase automatically. Any combination of these system fields is valid. These are the SECTION ACCESS fields:
    • ACCESS Defines the user access level with USER or ADMIN.
    USERID Contains the Qlik Sense user name.
    USER.EMAIL A string corresponding to a user email address
    GROUP Contains the group on Qlik Sense.
    OMIT Contains the field that is to be omitted for this user. Avoid this with key fields, as this will change the underlying data structure, breaking table associations.

Qlik Sense authorizes users utilizing the fields of the SECTION ACCESS:

  • If the table contains a USERID or USER.EMAIL field; Qlik Sense checks if the logon information matches any entry in the USERID or USER.EMAIL fields. If there is a match, the user is given access to the app.
  • If the table contains a GROUP field; Qlik Sense checks if the logged in user belongs to a group that matches an entry in the GROUP field. If there is a match, the user is given access.
  • The current user must match either USERID, USER.EMAIL or GROUP fields, or all of them, to get access to the app.
Section Access table example
SECTION ACCESS;
LOAD * INLINE [ACCESS, USERIDADMIN, ADMINISTRATORUSER, USERABC ];

SECTION APPLICATION;
LOAD... FROM... ;

Dynamic data reduction

Some of the data in an app can be hidden from the user, based on the logic implemented in the SECTION ACCESS. This allows you to hide:

  • Fields (columns) using the OMIT field in the SECTION ACCESS table.
  • Records (rows) by associating the SECTION ACCESS data with the SECTION APPLICATION data. Having one or more fields with common names in both sections allows Qlik Sense to reduce the data based on an initial selection by the SECTION ACCESS. After the selections have been made, Qlik Sense hides all data excluded by these selections from the user.

Note: The following wildcards are allowed in the SECTION ACCESS table:

  • An asterisk (*) defines all possible values when used in USERID or GROUP system fields. It can also be used in other SECTION ACCESS fields for data reduction. However, in this case, it represents all the listed values in the table.
  • A blank value represents all the values in a field when used for data reduction, or no fields are hidden when used in the OMIT field.
Section Access with dynamic data reduction example

The following example shows a SECTION ACCESS with data reduction:

  • The SA_LINK field associates the SECTION ACCESS table with a SECTION APPLICATION table for row reduction based on the DivisionName.
  • The OMIT field hides the DivTarget field for specific users.

For the USERID field, the user name includes the domain name, e.g. ABC. This would work predominantly in Qlik Sense on Windows environments.

The INTERNAL\SA_SCHEDULER user is required to enable reload of the script in a Qlik Management Console task.

SECTION ACCESS;
LOAD * INLINE [ACCESS, USERID, SA_LINK , OMITADMIN, ABC\AD1, ,USER, ABC\JCS, EMEA,USER, ABC\FRL, ,USER, ABC\IHX, SA, DivTargetUSER, ABC\IHX, NA, DivTargetUSER, ABC\BMN, *,USER, INTERNAL\SA_SCHEDULER, ,];

SECTION APPLICATION;
LOAD * INLINE [SA_LINK, DivisionName, DivTargetEMEA, EMEA, 15000NA, North America, 23500SA, South America, 19700APAC, Asia Pacific, 18900];

For the preceding script, the results for the ABC users are:

User Level Result
AD1 ADMIN All fields and all data including the 4 divisions.
JCS USER All fields, but only data associated with EMEA.
FRL USER Access denied to the app. A USER cannot have access to all data, only an ADMIN can.
IHX USER All fields excluding DivTarget and only data associated with North America or South America.
BMN USER All fields, but only data associated with the 3 divisions (EMEA, North America and South America).

Profiling data (SaaS)

Hub based data profiling tools

Source data files which are accessible within Qlik Sense can be analyzed with built-in profiling tools, available directly within the Hub. Different views are generated for complete analysis of the metadata associated with individual tables. Metrics describing field values include:

  • Count of distinct values (cardinality)
  • Sample values, most common values, and value frequency
  • Redundancies useful in identifying default or potential duplicate values
  • Counts of null, string, and numeric values
  • Information about value ranges including min, max, average, sum, and standard deviation

Access to this information aids developers and analysts in assessing optimal data model creation, as well as app creation strategies.

Profile Tile view

Profile tile view displays the most pertinent information for each field in a table, represented in a visual card format.

Each field has all values analyzed to determine the most likely field type, so that appropriate metrics can be generated and displayed. The metrics can be altered from numeric string format, to alter analysis from numeric averages to most frequent string values.

Profile List view

Profile list view displays a table with various statistical calculations for each field. The initial nine statistics are pre-selected by default, but various other aggregations and functions can be selected via a column selector option. In addition, three sample values for each field are displayed.

Data sample

Rather than the profile view of the fields, a data sample table can be selected. The sample displays 20 rows of value data for each field, to give developers and analysts a sense of the table contents.

Dataset properties

Entire tables can be tagged with properties for data compliance purposes. Properties such as GDPR, PCI and PII can be attributed to a data table and then marked with either Yes or No, to define whether the table complies with the standard or not.

Additionally, a single user-definable subject area field can be chosen for flexible table tagging.

Reloading from the Hub (SaaS)

Initiating an app reload in the cloud Hub

When developing app data models within the Data load editor, the reload button in the environment is referred to as an "interactive reload". Once app development has been completed, and data within the app requires updating, it is not intended that users consuming app data and performing analytics reload the app in this way. To manually update product-grade app data, users with permission can perform a reload of app data, directly within the Hub. This is achieved by using the action menu next to an app, inside the Hub and activating the reload.

Scheduling reloads

For apps which require data refreshes at a later point in time, it is possible to schedule reloads to occur against a calendar and clock. Such reloads can be triggered to run either once, or in a designated, repeating pattern at fixed intervals.

Intervals for repeating reloads include:

  • Hourly
  • Daily
  • Weekly
  • Monthly
  • Annually

In addition, once defined, a reload schedule can be temporarily deactivated to prevent it from being triggered, such as when app development is continuing or maintenance is required.

Tenant Administrators can review all reload schedules across all apps and delete them from the central Management Console, if required.

App reload history

When an app is either manually reloaded in the cloud Hub, or a scheduled reload occurs, a log of the reload is created against the app. This log can be viewed for the specific app to check for warnings or other issues relating to reloads.

Evaluating app performance (SaaS)

Performance evaluation within the cloud Hub

During and after development in Qlik Cloud, you can evaluate the performance of apps to ensure the front end experience for users is optimized. An automated performance evaluation routine can be initiated from an app in the Hub, which subsequently runs through numerous tests against publicly-accessible sheets and object within the app.

The routine requires that all sheets and objects which need evaluation have been made public, otherwise the process is not effectively analyzing the app.

In addition, if SECTION ACCESS security has been implemented on an app to reduce the data available for the user initiating the evaluation, it is only that data which can be utilized within the test.

Keep in mind that the results are provided as guidance and cannot guarantee actual user-perceived performance in production environments.

Metrics calculated

The numerous metrics which are calculated as part of the evaluation are beyond the scope of this course, but for reference, include:

  • Object calculation caching
  • Single-threaded performance
  • Cached vs non-cached render time of objects
  • General memory allocation

Due to the nature of the rigorous performance tests, evaluations can take a protracted amount of time. Therefore, it is possible to schedule notifications when app evaluation is complete.

Advanced analytics integration in Qlik Sense

Data analytics spectrum

Analytics capabilities are typically categorized as:

  • BI / Reporting (descriptive, diagnostic business needs)
  • Visual Analytics (diagnostic business needs)
  • Advanced Analytics (diagnostic, predictive, prescriptive, and optimization outcomes needed by the business)

Together, these categories create the analytics spectrum with various technologies to help support decision-making across the organization.

(Click on the image below to enlarge)

Qlik takes a platform approach so it can deliver a full spectrum of business intelligence and visual analytics capabilities to organizations. The platform provides a unified framework for governance, security, and scalability. With the integration of Advanced Analytic Integration (AAI) and an open API strategy, Qlik Sense is ideal for delivering advanced analytics. It can offer the interactive, associative exploration of governed models built by the Data Scientist community.

Gather more information by reading the article "Four Types of Business Analytics to Know".

Advanced Analytics in Qlik Sense

Qlik Sense uses two different ways to perform Advanced Analytics tasks:

  • Native, built-in capabilities:

    • Qlik Associative engine contains a long list of statistical functions, interactive visualizations, and scripting for many Advanced Analytics use cases.

  • Advanced Analytics Integration (AAI)

    • AAI expands Qlik Sense's existing capabilities and allows executing Advanced Analytics tasks such as predictive, prescriptive, or optimization analytics.

Advanced Analytics Integration (AAI)

Advanced Analytics Integration allows users to leverage the power of external analytics engines and machine-learning algorithms, such as R, Python, Matlab, Spark, etc., directly from Qlik Sense.

  • Provides direct integration with third party, Advanced Analytics engines through server-side extension APIs.
  • Allows data to be directly exchanged between the Qlik Associative engine and external tools during analysis.
    • Leverages the Qlik Associative Model to pass relevant data based on user context.
  • Provides full integration with Qlik Sense expressions and libraries.
  • Allows connectors to be built for any external engines.
  • Contains open-source connectors (created by Qlik) for R and Python.

How AAI works

Chart expressions

Advanced Analytics Integration in Qlik Sense happens as the user explores information. Data is passed between the Qlik Associative engine and third party tools in a more real-time manner than with other tools, which rely more on batch / pre-processing approaches for larger data sets.

When a user makes a selection, a number of steps take place:

  1. A search is performed within an app and associated / unrelated values are determined by the Qlik Associative engine.
  2. The hypercube behind the integrated visualization is dynamically updated by the Qlik Associative engine.
  3. The reduced, in-context (associated) data set and relevant script are sent to the external engine.
  4. The external engine runs calculations and sends the resulting data back to the Qlik Associative engine.
  5. The Qlik Associative engine integrates the resulting data with the hypercube for the integrated visualization.
  6. The combined data set is visualized for the user.

Script

During the script execution, the Qlik Associative engine passes R and Python script commands and functions to the third party engines. The engines perform the additional, Advanced Analytics calculations and return the results back to the Qlik Sense engine, so that they can be used in an associative way.

AAI architecture

The AAI architecture consists of three main components:

  • Qlik Sense
    • Qlik Cloud or Qlik Sense Client Managed
    • It exposes the API to the Connector.
  • Connector
    • gRPC A light-weight, open-source, remote procedure call (RPC) protocol, that enables client and server applications to communicate transparently in a network environment and send data back and forth between systems. server
    • Qlik Sense and third party engine both act as clients to this server.
    • Calls the API that is exposed by Qlik Sense
    • As R is not natively supported by gRPC server, the SSEtoRserve plug-in needs to be installed to allow communication between R and gRPC.
  • Third Party Engine
    • Typically R or Python, but can be anything with an open API and called from the gRPC server.
    • Server-Side Extension (SSE) APIs are utilized for this additional functionality.
