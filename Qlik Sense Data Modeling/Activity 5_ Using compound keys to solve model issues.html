Data Modeling for Qlik Sense

Activity 5: Using compound keys to solve model issues

Summary:

  • Load shipments data from the ABC_2025 database and review the issues caused in the data model.
  • Create compound keys in the Shipments and OrderDetails sections to provide one, single field to associate the tables upon.
  • Comment out additional fields which cause synthetic keys and circular references.

Module: Generating data

Qlik Sense app: ABC Sales Data

Challenge

In the ABC Sales Data app, load a new table and observe the issues in your data model. To achieve this:

  • Navigate to the end of the Shipments script section.
  • Load the shipments table from the ABC_2025 database and label it Shipments. Add appropriate commentary.
  • Load your data and check the data model to verify that the table has been loaded; observe the issues that have appeared in your data model.
  • Navigate to the Shipments script section and locate the Shipments table.
  • Create a composite key based on OrderID and LineNo:
    OrderID &'|'& LineNo AS OrderLineKey,
  • Comment out any fields from tables that you no longer need.
  • Copy the composite key you just created to the OrderDetails table to create a new association.
  • Load your data and verify that the synthetic key between the Shipments and the OrderDetails tables is gone.
  • In the Shipments table, use a WHERE EXISTS clause to limit the records you are loading to only those which contain OrderID in the dataset.
  • In order to optimize the script, use the AutoNumber function with the composite keys you created for both Shipments and Orders tables. Observe the difference by previewing the OrderLineKey field.
  • In the Shipments table of your script, comment out the fields that are repeated in the Orders table. Leave only the OrderLineKey key and the ShipmentDate, as they are the only fields you need to keep in the data model.
  • Load your data and verify there are no more issues in your data model.

Hints

  • Three synthetic keys have been created in your data model because the new table has multiple fields in common with Orders and OrderDetails tables.
  • Comment out the fields OrderID, LineNo and ProductID from the Shipments table ONLY as they are no longer needed after the composite key.
  • The following three fields must be commented out from the Shipments table: CustomerID, EmployeeID, and ShipperID. These fields also exist in the Orders table, so the data is available through the associations.
  • Eliminating the synthetic key will also eliminate the loop (circular reference) in your data model.

 

Part 1 - Load the Shipments table

Step-by-step guide
  1. Navigate to the end of the Shipments script section.
  2. Use the ABC database connection to load the shipments table and make sure all the fields are selected.
  3. Add a suitable comment before the LOAD statement and relabel the table [shipments]: whilst capitalizing the "S".
  4. Remove the LIB CONNECT statement between the Shippers and Shipments table loads.
  5. Load the data and examine the synthetic key warnings.
  6. In the Data model viewer, notice a synthetic table was generated. The $Syn 3 Table contains 3 synthetic keys with common fields between the OrderDetails, Orders, and Shipments.
    There is one synthetic key for each table because the common fields vary:
    • $Syn 1 associates the synthetic table with OrderDetails. The common fields are: OrderID, LineNo, and ProductID.
    • $Syn 2 associates the synthetic table with Orders. The common fields are: CustomerID, EmployeeID, ShipperID, and OrderID.
    • $Syn 3 associates the synthetic table with Shipments. The common fields are: OrderID, LineNo, ProductID, CustomerID, EmployeeID, and ShipperID.

    In the scenario you are currently working on, there is a business rule that says each product can be shipped separately. So, even if you have products that belong to the same OrderID, they can still be shipped as individual items.

    This means that there is a one-to-one relationship between OrderDetails and Shipments. The field ProductID gets the values directly from OrderDetails. There is no need to have it twice in the data model. Therefore, remove the ProductID field from Shipments, in the next part of the activity.

Part 2 - Create a composite key

Step-by-step guide
  1. Navigate to the Shipments script section and locate the Shipments table.
  2. Create a composite key field, based on OrderID and LineNo using “&” to concatenate the fields and the pipe (|) as separator. Name the key field OrderLineKey:
    OrderID & '|' & LineNo AS OrderLineKey,
  3. Comment out the fields OrderID,  LineNo, and ProductID in the Shipments table. Since you have now found an alternative way to associate the two tables, there is no point of keeping those fields twice in the data model. This is necessary to avoid the synthetic key. Those fields still remain in the OrderDetails table.
  4. Copy the composite key you just created and paste it to the OrderDetails table. Leave the fields OrderID and LineNo unchanged.
  5. Load the data and observe the warning messages in the load progress. You get a warning for one synthetic key and one error due to a circular reference because one or more loops have been found in the data model. Notice that before, there were 3 synthetic keys. Now, there is just one remaining.
  6. Switch to Analyze mode, create a new sheet and name it Composite key.
  7. Enable edit mode on the new sheet.
  8. Add the following dimensions to the sheet as Filter panes: OrderID, LineNo, and OrderLineKey.
  9. Disable the edit mode in the sheet and select one value in the OrderID field. Review the associated values (white) in the OrderLineKey field.
  10. Clear the selection. Choose an item from the top of the OrderLineKey filter pane, e.g. 1|1. There are some values in the field that do not associate with any OrderID and LineNo. Why is this? In a previous activity, you used a WHERE clause in the Orders and OrderDetails tables. The Shipments table loads all the orders instead.
  11. Clear the selection and return to the Data load editor.
  12. In the Shipments script section, add the following WHERE clause before the semicolon in the preceding LOAD statement:
    WHERE EXISTS(OrderID, OrderID);
  13. Apply the AutoNumber function to get a numeric representation for the OrderLineKey field. In the Data load editor, change the OrderLineKey line in the OrderDetails and Shipments tables to match the following script:
    14. AUTONUMBER(OrderID & '|' & LineNo, 'OrderLine') AS OrderLineKey,

    Note the second parameter in this function; OrderLine is optional in this case, as there are no other composite keys using the AutoNumber function. It is added here as a best practice.

  14. Load the data again and open the Composite key sheet. Check the values in the OrderLineKey field. They are now sequential numeric values.

Part 3 - Eliminate the second synthetic key

Step-by-step guide
  1. In the Data model viewer, observe your data model. The synthetic key and circular reference exist because there is more than one common field between the tables Shipments and Orders. In addition, there are connections from Orders to OrderDetails to Shipments to $Syn 1 Table, and back to Orders again.
  2. Navigate to the Shipments script section and comment out the three fields CustomerID, EmployeeID, and ShipperID from the Shipments table.
  3. Ensure that the last non-commented field in the LOAD statement, ShipmentDate, is not followed by a comma.
  4. Load your data and notice that there are no warnings or errors during the reload.
  5. In your data model, verify that the circular reference and the synthetic key have been eliminated.
