Data Modeling for Qlik Sense

Activity 7: Evaluating app performance (SaaS)

Summary:

  • Initiate app performance evaluations.
  • Analyze the evaluation results.
  • Understand the impact of Section Access security with respect to performance analysis.

Module: Exploring security and advanced concepts

Qlik Sense app: ABC Sales Data / ABC Sales Data_eval / ABC Sales Data_SA3

Challenge

Use the built-in SaaS app performance evaluation feature, to:

  • Analyze app memory consumption.
  • Understand how different tables and fields consume different levels of resource.
  • Identify how to dig deeper into the analysis results.
  • Appreciate the impact which SECTION ACCESS security has on performance evaluation.

Hints

  • The performance evaluation tool is a feature to help guide Developers to potential issues with app development.

 

Step-by-step guide
  1. Ensure you are logged in to Qlik Cloud with your primary Qlik ID.
  2. Navigate to the Personal space and use the Types drop-down to filter by Apps, if required.
  3. Duplicate the ABC Sales Data app.
  4. Use the More actions ellipsis and the Rename option, to rename the copy to ABC Sales Data_Eval.
  5. Click the Save button, then utilize More actions for the new app and select Details.
  6. Select the Performance evaluation tab.
  7. The list should be empty, so click Evaluate performance to queue up an app health check.
  8. After a short duration, the evaluation should be complete, and an entry appears in the list, marked as Complete.
  9. Click on the new entry to investigate the findings.
  10. The evaluation dialog displays an Overview of the findings, indicating there is just one public sheet in the app, containing ten objects which it evaluated. Switch to the Details tab.
  11. Expand Top 5 tables with the largest memory allocation.
  12. The findings indicate that the OrderDetails table consumes the most memory of all tables, followed by the Orders table. This is logical, given the nature of these two tables as fact detail tables. Collapse the view.
  13. Expand Top 5 fields with the largest memory allocation.
  14. According to the analysis, the field %OrderKey accounts for the greatest memory allocation in the data model. This also makes sense, as the field is a deliberate creation to hold unique values, despite the use of the AutoNumber function. Collapse the view.
  15. Expand Top 5 objects with the largest cached load time.
  16. The table displays a list of object IDs and the associated time to load them. For the top item, click the button to Open in new tab.
    17. A new browser tab opens and displays the publicly-available Employees sheet. The relevant object is briefly indicated with a colored-border to draw attention to it.
  17. Close the browser tab to return to the evaluation details. Then, click the Done button to close the dialog.
  18. Go Back to the app list and select the DMQS space.
  19. Locate the ABC Sales Data_SA3 app with your own username suffix, click the More actions menu and go into the Details.
  20. Select the Performance evaluation tab.
  21. Click Evaluate performance to queue it up.
  22. Once it completes, click on the new entry to open the details.
  23. Observe the details on the Overview sheet.
    24. Note: The app still contains one public sheet with ten public objects to evaluate. However, the SECTION ACCESS utilized within the app can hamper the evaluation of the performance, as it is no longer a true reflection of the different data users may see.
  24. Use the Done button to close the log, and the Back link to return to the Hub.
