Aggregation tables

Sometimes, you have to aggregate values from a lower level of granularity and store them in an aggregate (or summary) table for better performance or for the creation of complex calculations during reloading.

To group or aggregate data, use the GROUP BY clause in the LOAD statement. GROUP BY is a clause used for defining the fields over which the data is to be aggregated (grouped).

Example:

Assume you have the following table, which is a simplified version of the OrderDetails table in your data model. This table is an order-line level table, illustrating one row for every order line and multiple rows for the same OrderID.

OrderID LineNo LineSalesAmount
1 1 30
1 2 20
1 3 50
2 1 10
2 2 20
3 1 40

The aggregation table is a new, order-level table which contains unique OrderID values and a calculated field, OrderAmount (Sum(LineSalesAmount)), that holds the total sales amount for every order. The new table has one row for every OrderID value, as shown in this image:

 

OrderID

OrderAmount

Sum(LineSalesAmount)
1 100
2 30
3 40

To create this table, the following script can be used:

LOADOrderID,SUM(LineSalesAmount) AS OrderAmountFROM [lib://<ConnectionName>/Table1.csv]
GROUP BY OrderID;

In this example, the LineSalesAmount field is aggregated over each order from the source table providing a value for OrderAmount.

Cross tables

A cross table is a two-way table consisting of columns and rows. It is also known as a pivot table or multi-dimensional table, and often looks like this:

Year Jan Feb Mar Apr May
2016 45 65 78 12 22
2017 11 23 22 22 65
2018 65 56 22 45 15
2019 45 24 32 73 86
2020 45 56 35 56 11

If this table is simply loaded into Qlik Sense, the result is one field for Year and one field for each of the Months. However, it is more useful to have three fields generated, one for each of the header categories; Year, Months, and one for the data values inside the matrix.

This is how Qlik Sense loads the data as seen from the table image:

This is how the table must be loaded:

 

In Qlik Sense, you can achieve this by adding the CROSSTABLE prefix to the LOAD statement. There are three types of fields that can be generated from a cross table:

  • Qualifier field — fields in the first n columns that are already structured as required. In the example above, there is just one, Year.
  • Attribute field — the field that results from the column headings, that are not Qualifier fields, merged into a single column. Instead, they are repeated for each row of data; in this case, Months.
  • Data field — the field that contains all of the data values from the matrix when they have been merged into a single column; in this case, SalesAmount.
Note: Alternatively, you can use the Data manager and the UnPivot F option to load a table as a cross table. In the accompanying activities, you are going to use the script to manually convert the table as part of the transformation process.

The syntax for CROSSTABLE is: 

CROSSTABLE (attribute field, data field [ , n ] )
( Load statement | Select statement )

The statement for loading the example table can be written as:

CROSSTABLE (Months, SalesAmount, 1)
LOAD * FROM [lib://dataconnection/sales.csv]
(ooxml, embedded labels, table is Sheet1);

This creates a table in the data model with the three fields: Year, Months, and SalesAmount.

Link tables

When loading data, certain scenarios can occur that require storing fact type of information in different tables, such as transactions separated from budgets. Each fact type may need to be joined with multiple, common dimensions (such as Time and Product). This causes loops in the data structure. Create Link tables to store the common dimensions in such scenarios.

Link tables:

  • Combine information that has different levels of detail or frequency.
  • Are used to solve the issue of many-to-many relationships between tables. They are best used when the metrics being measured do not always have the same granularity. An example of this is when one metric is aggregated over Year and Month, and another is based only on Year.
  • Are useful when linking information that is not otherwise automatically linked with Qlik Sense associative logic.
  • Can be considered handmade, synthetic keys.
  • Allow the Data Architect to keep the fact tables separated from each other. Alternatively, the fact tables would have to be concatenated together and the usage of Set Analysis syntax would be necessary.
  • Are commonly used to eliminate the need for multiple calendar tables.

Using concatenation to create the Link table

Concatenation:

  • Is often used when creating a Link table and when the tables contain the same common fields.
  • Combines tables with common dimensions into the Link table, creating all possible combinations between the common values.
  • Appends records to the Link table, only using the common fields (common dimensions) from the tables that create the Link table.

When creating a Link table, concatenation can be used if the tables have different granularity of information. But, a JOIN operation is preferable because the output records are linked on a record-by-record basis. There is no risk of duplicating records in a Link table since all the fields are dimensions.

How to create a Link table

The basic steps to create a Link table are:

  1. Load the fields from all the tables.
  2. Rename the fields that cause synthetic keys or circular references in your data model.
  3. Create composite keys to use as key fields in the tables.
  4. LOAD DISTINCT...RESIDENT from the tables to create the Link table, and rename the fields back to their initial names. The new table contains all the renamed fields from the above tables.
  5. Add the composite keys you created on the other tables.
  6. Concatenate all the tables. Always use LOAD DISTINCT for the Link table.
  7. To be able to distinguish the source of each record, add a flag field that contains the source information to the Link table, such as, FromTable.
  8. DROP the tables and fields you don't need any more.

Although these steps cover the majority of the cases, there might be occasions where a different sequence of steps needs to be followed.

Data classification

Data classification is the process of organizing data into categories for its most effective and efficient use. A well-planned data classification system makes essential data easy to find and retrieve.

Qlik Sense handles the specification of data in various ways. The most common method is by using IntervalMatch, but the CLASS and IF functions are also an option.

Class

CLASS is a script and chart aggregation function that you can use to create buckets of information.

This function allows you to group your dimension values in buckets fixed to the same width.

The CLASS syntax is:

CLASS(expression, interval [ , label [ , offset ]])

Example:

CLASS( 23, 10, 'x', 5 )

This returns '15<=x<25', as the number 23 falls in the bucket 15-25 when buckets are 10 wide and they start the series with 5.

For a solution with varying size buckets, consider IntervalMatch or nested IF F statements.

IntervalMatch

The IntervalMatch function is used to create a table, matching discrete numeric values to one or more numeric intervals and, optionally, matching the values of one or several additional keys.

The IntervalMatch syntax is:

IntervalMatch (matchfield [,key1 [,key2,..]])
Load | Select statements

The IntervalMatch prefix must be placed before a LOAD or a SELECT statement that loads the intervals. The field containing the discrete data points, matchfield and additional keys, must already have been loaded into Qlik Sense before the statement with the IntervalMatch prefix.

This prefix:

  • Transforms the loaded table of intervals and keys to a table that contains an additional column with the discrete numeric data points (matchfield).
  • Expands the number of records so that the new table has one record per possible combination of discrete data point, interval, and value of the key fields. The intervals may be overlapping and the discrete values are linked to all matching intervals.
  • Creates a synthetic table. When applying this prefix, consider using a LEFT JOIN between the IntervalMatch table and the source of this table to avoid synthetic keys remaining in the data structure.
